# 我给两个 AI 写了一部"宪法",然后把它做成了一行 `npx`

## 故事得从一个尴尬说起

如果你同时用 Claude Code 和 opencode 指挥一堆 code agent 干活,大概率体会过这种尴尬:同样的坑,换个项目、换个 agent,又踩一遍。Ant Design 那个该死的两字中文按钮会自动塞进一个空格,害得 Playwright 选择器匹配不上;GORM 里 `false` 是 bool 零值,Create 的时候被悄悄吞掉;部署完上线一看,页面纹丝不动——因为 `git archive HEAD` 只打包已提交的文件,你那堆没 commit 的改动根本没上船。

这些教训我都交过学费。问题是,学费交了,知识却散落得到处都是:有的躺在某个项目的 `CLAUDE.md` 里,有的写进了 `.claude/projects/*/memory/` 的某个 feedback 文件,有的只活在一两个月前某次会话的上下文里,会话一关就烟消云散。每开一个新项目、每拉起一个新 agent,等于让它从一张白纸重新开始当新手。

我想干的事很简单:**把这些散落的、隐性的经验"蒸馏"出来,变成一份能被多个 agent 共享、能控制它们稳定交付的"操作系统"。** 一份给人看也给机器读的工程宪法。

## 蒸馏:从十五个项目里熬出一锅"汤"

第一步是考古。我让 agent 去扫了一遍家底——大约 15 个带 `CLAUDE.md`/`AGENTS.md` 的项目、66 个记忆文件、上百次会话记录、十几个自定义技能,文本量两百多 KB。然后沿着八个维度去提炼:用户画像、跨语言的核心原则、跟 AI 交互的"话术"、按技术栈分类的踩坑、测试方法论、文档规范、部署运维、多 agent 编排。

(顺带说个小插曲:我开工的时候发现工作目录里有个"同事"正在实时往里写分析文件——文件时间戳每隔几秒就变一个。这是典型的并发协作场景,我没去抢他的活,而是另起炉灶把最终交付物放进单独的子目录,只读引用他的分析层。并发安全这根弦,从第一分钟就得绷着。)

蒸馏出来的东西,我没有一股脑塞进一个巨型 `CLAUDE.md`——那玩意儿每次会话都要烧一遍 token,还特别容易过期。我用的是**薄宪法 + 按需分层**:最顶上是一层极薄的元规则,薄到一屏能看完,永远常驻;具体的细节往下沉,变成"碰到才加载"的分栈参考和能自动触发的技能。

而且我给每一条规则都打了标签:🟢 是放之四海皆准的工程原则(TDD、举一反三、声称完工前必须真跑一遍看到绿、别信子 agent 的自我汇报),🟡 是"这个人/这个团队的偏好"(config-driven 零 DDL 迁移、中文枚举值不准翻译、遇到选择就选推荐项一路跑到底)。这一步特别关键——它防止后来 fork 的人把别人的个人习惯当成宇宙真理。

## 当"文档"开始想变成"程序"

本来到这儿就该收工了。但需求开始长腿:

能不能让这套机制同时管住两套系统,而且**从每个工具自己的全局 md 文件作为顶层入口**?能不能把那些技能注册成真正能被自动触发的 skill,而不是靠我每次记得去念咒?那些"不可绕过"的纪律——比如完工必验证、部署前必 commit——能不能做成确定性的 hook,绕过模型的临时判断,硬卡住?

最后,临门一脚:**能不能把这整套东西做成一个能发布的安装包,任何人在 Windows 或 Linux 上拿到它,它会自动探测系统里 Claude 和 opencode 装在哪,然后把规则全局注册进去——两个工具都装了,就两个都生效。**

到这一步,"文档"已经按捺不住,要变成"程序"了。

定技术选型时几乎没有悬念。Claude Code 和 opencode 本身都是 npm 分发的,用户机器上必然有 Node;那些 hook 脚本本身就得是跨平台的 Node 脚本;安装逻辑无非是探测路径、合并 JSON、复制文件。于是答案明摆着:**一个发布到 npm 的零依赖 Node CLI**,`npx` 一把梭,Windows / Linux / macOS 通吃,不用为每个平台编译一份二进制。

研究两个工具的接入点时,有几个发现让设计一下子清爽了。最妙的是:**opencode 原生就会去读 `~/.claude/skills/` 目录**。这意味着技能装一份,两个工具都认——双工具并存时根本不用装两遍。Claude 这边走 `settings.json` 的 hooks,opencode 那边没有配置级 hook、只能靠 JS 插件,那就给它单独写个插件。还有个硬约束:skill 的名字必须是 ASCII 小写加连字符(opencode 的正则卡得死),所以我那六个中文名的技能全得改成 `recursive-sweep`、`deploy-to-prod` 这样的英文名,中文留在正文里。

## 真实世界永远比设计文档脏

代码写完,我没说"应该能跑"——这恰恰是我写进宪法里第一条要禁止的话。我把它真装进临时目录跑了一遍,又把三个 hook 一个个喂数据测。然后,真实世界开始还以颜色。

测 `ensure-commit` 这个"部署前没 commit 就拦住"的钩子时,它死活不拦,一路放行。查了半天才发现是我自己测试命令里的锅:我手写的 JSON 里 `"cwd":"D:\acr-test"` 那个 `\a` 是非法的 JSON 转义,钩子解析失败、安全降级放行了。换成正斜杠路径,立刻就拦住了,还吐出一句"未提交的改动不会被打包,这是'部署后页面没变化'的头号元凶"——那句话本身,就是从踩过的坑里蒸出来的。

更阴险的是另一个 bug。我用受控标记块往用户的 `CLAUDE.md` 里塞内容,标记长这样:`<!-- BEGIN ai-coding-rules (managed) -->`。结果重装一次,块就重复一次;卸载还删不掉。盯着正则看了好久才反应过来——那对括号 `(managed)` 在正则里是个捕获组,匹配的是裸字符串 "managed",可文本里实实在在带着括号,于是匹配永远失败,既删不掉旧的、又一直往后追加新的。一个 `escapeRegExp` 转义就解决了。修完重测:装两次,块只剩一个;卸载干净,用户原本写在文件里的私人笔记原封不动。这才算数。

## 发布:一场和 npm 的小型拉锯战

代码过关了,发布环节又是连续的"惊喜"。

先是 `npm login` 弹出来一看,登的居然是 `registry.npmmirror.com`——国内镜像,只读,根本发不上去。改成显式指定官方源。然后 2FA 拦路,说要么 OTP、要么带"绕过 2FA"的细粒度 token。我让用户建了个细粒度 token,结果第一个建成了只读,发布被拒;换成"读写全部包"的才过了 2FA 这关。刚松口气,又撞上 `403:名字 ai-coding-rules 跟已有的 aicoding-rules 太像`——npm 防钓鱼。于是加上命名空间,`@decklijia/ai-coding-rules`,`0.1.0` 总算落地。

后面接上 GitHub:`gh` 已经认证成 `l17728`,建仓、推送、把 `repository` 指对。再往后,把默认分支从 `master` 换成 `main`,加上一条**免 token 的自动发布流水线**——用 npm 的 Trusted Publishing,基于 OIDC,GitHub Actions 直接拿身份令牌发版,机器上、CI 里都不用存任何长期 token。

这条流水线第一次跑,又挂了,报 404。OIDC 的 404 其实是"身份对不上"的伪装。翻配置一看——npmjs 上 Trusted Publisher 那栏,组织名填成了 `l117728`,比真名 `l17728` 多了一个 `1`。就这一个字符。改对、重跑,绿了,`0.1.4` 由 GitHub Actions 用 OIDC 悄无声息地发了出去,还自带 provenance 出处证明。

从最初的考古蒸馏,到代码、测试、踩坑、发布,这一切是在**一个连续的工作会话里一气呵成的**,前后大半天。最终一路 `0.1.0` 滚到 `0.1.8`,八个版本,大多是文档和体验的打磨。

## 它最终长成了什么样

成品是个克制的小东西:三十七八个文件,打包 28 KB,解开不到 80 KB,**零运行时依赖**——纯 Node 内置模块。里面装着四层:一份薄宪法、十六个会自动触发的技能(八个分栈参考 + 六个流程 + 一个"盯死项目 md"的冷启动守卫 + 一个驱动持续再蒸馏的闭环技能)、三个 Claude 钩子加一个 opencode 插件、还有一份会随安装落到本地的使用手册。

## 怎么用:一行命令的事

最简单的路子,一次性,不留痕:

```bash
npx @decklijia/ai-coding-rules install
```

它会自己探测你机器上的 Claude Code 和 opencode(opencode 在 Windows 上的配置目录因版本而异,它会调 `opencode debug paths` 去问准),然后把规则全局注册进去。两个都在,就都装。一个都没有?它不会默默失败,而是告诉你怎么装,并提示可以加 `--force` 先把规则预置好,等工具装上自动生效。

想反复用、或者喜欢敲短命令的,先把工具常驻:

```bash
npm i -g @decklijia/ai-coding-rules
ai-coding-rules install
```

这里有个新手常困惑的点值得点一下:`ai-coding-rules install` 里的 `install` **不是 npm 的命令**,而是这个工具自己的子命令——就像 `git commit` 里 `git` 是程序、`commit` 是动作。`npm install` 是"把工具装到机器上",`ai-coding-rules install` 是"运行工具、把规则部署进 Claude/opencode",两个 install 八竿子打不着。

动手前心里没底?加 `--dry-run`,它只打印将要做什么,一个字节都不落盘。担心它糟蹋你现有的配置?不会——它往 markdown 里写东西用的是带标记的受控块,改 JSON 只动自己那几个键,碰过的文件一律先备份成 `*.bak-ai-coding-rules`。想全身而退,`ai-coding-rules uninstall` 连 `settings.json` 里它加的那几条钩子都给你摘干净。装完想读手册,`ai-coding-rules manual` 给你路径。

还有个我挺喜欢的小开关:在某个项目根放一个 `.ai-coding.json`,写上 `{ "verify": "npm test" }`,那个"完工必验证"的 Stop 钩子就会在你想收工时真去跑一遍测试,挂了就拦住不让走(还防了死循环)。把"应该测过了"这种自我安慰,变成确定性的闸门。

## 最妙的部分:一个会"自我繁殖"的工具

这个项目我最得意的,是它能复制自己的"生产方式"。

发布到 npm 的,只是**工具本身**——你拿它去部署我蒸馏好的规则。但如果你想要的是**你自己的**规则集呢?那就 `git clone` 这个仓库。里面有一份 `AUTHORING-RULES.md`,把"如何从你自己的会话、记忆、文档蒸馏出规则"这套方法论写清楚了,还附了一段可以直接喂给 agent 的 Runbook——你把原始数据的位置告诉它,它就能照着蒸馏、归到四层、写进 `content/` 和 `hooks/`,然后跑一句 `node bin/cli.js validate` 自检。

这个 `validate` 是质检闸门,它会查技能的 frontmatter 合不合规、结构全不全,还会**扫描密钥**——因为原始会话和配置里常常躺着真实的 API key(我亲眼在某个 `opencode.json` 里见过),绝不能让它们被蒸进规则发出去。

换句话说,这套"元生产"能力专门留在 git 仓库里(`docs/` 被排除出 npm 包,保证发布出去的就是纯粹的工具);谁想造自己的版本,clone 下来照着干就行。仓库和 README 里我反复提醒:fork 之后**必须换成你自己的身份**——你的 npm scope、你的仓库、你自己的 Trusted Publisher,别误发到我的命名空间(也发不进去,你没权限)。整条链路因此闭合成一个环:新教训 → 再蒸馏 → 升版 → 重发,由那个 `system-improvement-loop` 技能驱动。

## 它到底好在哪

往回看,这个小工具的价值不在代码量,而在它把几件容易被忽视的事做对了。

它**真正跨了两套系统**,而不是只伺候一家——还顺手发现 opencode 会读 Claude 的技能目录,省掉了重复注册。它**分层而薄**,该常驻的常驻、该按需的按需,不拿一堆 AntD 的冷知识去糊你每一次会话的上下文。它把"纪律"分了三个硬度:能讲道理的写进宪法,能自动触发的做成技能,真正不可绕过的(部署前 commit、完工验证)才升级成确定性的 hook——因为只有 hook 是不看模型脸色的。它**从不偷偷糟蹋你的文件**:受控块、键级合并、自动备份、可干净卸载。它**零依赖**,一个 `npx` 就跑,CI 发布**全程不碰 token**。

但我自己最看重的,是它没把"经验"当成一次性的、私有的东西。它把蒸馏的方法连同工具一起开源了,还配了校验闸门和防身份串号的提醒。它不只是给你一套规则,而是**把"如何生产规则"这件事本身也交到了你手里**。

所以如果你也受够了让每个新 agent 从头当新手,不妨试一句:

```bash
npx @decklijia/ai-coding-rules install
```

源码、方法论和那张架构图都在 [github.com/l17728/ai-coding-rules](https://github.com/l17728/ai-coding-rules)。拿去,改成你自己的,然后让你的 agent 们,从此都带着记忆开工。

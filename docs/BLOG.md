# 周末闲来无事,我把自己"蒸馏"了一遍

周末两天,我做了一件事：把自己过去一年用 AI 写代码积累的经验——踩过的坑、总结的纪律、各技术栈的注意事项——全部提炼出来,做成了一套可安装的工程规则和自动触发的技能包。然后把它包成了一个 npm 工具,一行命令就能装到 Claude Code 和 opencode 里,让任何新拉起的 AI agent 都不再是新手。

**它能干什么？** 装上之后,AI agent 会自动遵守一些工程纪律：部署前必须 commit、完工前必须真的跑一遍测试、发现一个 bug 会自动搜遍全库找同类问题。它还会根据你用的技术栈自动加载对应的知识——Ant Design 的坑、GORM 的零值陷阱、Electron 的打包注意事项,不用你每次手动提醒。

**怎么装？** 一行命令：

```bash
npx @decklijia/ai-coding-rules install
```

它会自动找到系统里的 Claude Code 和 opencode,把规则注册进去。两个都有就都装,只装了一个就只装那一个。想先看看它会做什么,加 `--dry-run`。想卸载,`ai-coding-rules uninstall` 干净移除。

下面是这件事的来龙去脉。

---

## 起因：学费交了,知识却留不住

如果你同时用 Claude Code 和 opencode 写代码,你可能遇到过这种事：同样的坑,换个项目、换个 agent,又踩一遍。

Ant Design 两个中文字的按钮会自动塞空格,Playwright 选择器就匹配不上了。GORM 里 `false` 是 bool 的零值,Create 的时候被悄悄吞掉。部署完上线一看页面没变化,折腾半天发现 `git archive HEAD` 只打包已提交的文件,没 commit 的改动根本没上去。

这些教训我都交过学费。但学费交了,知识却散落得到处都是：有的躺在某个项目的 `CLAUDE.md` 里,有的写在记忆文件里,有的只存在于某次会话的上下文里,会话一关就没了。每开一个新项目,等于让 AI 从零开始当新手。

我想做的事很简单：把这些经验整理出来,变成一份能让多个 agent 共享的规则。人能看懂,机器也能读。

## 蒸馏：从十五个项目里提炼规则

我让 agent 扫了一遍家底——大约 15 个带配置文件的项目、66 个记忆文件、上百次会话记录、十几个自定义技能,文本量两百多 KB。然后从几个维度提炼：哪些原则是通用的、哪些是特定技术栈的坑、测试怎么做、部署要注意什么、多个 agent 怎么协作。

整理出来的东西,我没有一股脑塞进一个巨型配置文件——那种文件每次会话都要读一遍,浪费 token,还容易过时。我用的是分层的思路：最顶上是一层极简的元规则,一屏能看完,永远常驻；具体的细节按需加载,碰到相关场景才读。

每条规则打了标签。🟢 是通用的工程原则——写完代码必须真的跑一遍测试,不能只说"应该没问题"；agent 说"搞定了"不能直接信,得去看它实际改了什么。🟡 是个人偏好——中文枚举值不要翻译、遇到选择题就选推荐项。这个区分很重要,防止后来的人把个人习惯当成金科玉律。

## 从文档到程序

本来整理完就该收工了。但需求在慢慢长腿。

能不能让这套规则同时管住两个工具？能不能把经验注册成自动触发的技能,不用每次手动去翻？那些"绝对不能跳过"的步骤——部署前必须 commit、完工前必须跑测试——能不能做成硬性检查,不让 AI 偷懒？

最后一步：能不能做成安装包,别人拿到手跑一下就行？

技术选型没什么悬念。两个工具都是 npm 分发的,用户机器上肯定有 Node。做一个零依赖的 Node CLI,`npx` 一行搞定,全平台通吃。

研究接入方式时有个意外发现：opencode 原生就会读 Claude 的技能目录。技能装一份,两个工具都认,省了重复安装。Claude Code 通过 `settings.json` 注册钩子,opencode 没有配置级 hook,就给它单独写了个 JS 插件。

## 几个印象深刻的坑

我在规则里写了一条"不允许说'应该能跑',必须真的跑一遍"。所以写完代码,我把它装进临时目录实际测了。然后就开始踩坑了。

`ensure-commit` 钩子的作用是部署前没 commit 就拦住。测试时它死活不拦。查了半天,发现是我自己写的测试 JSON 里路径 `"cwd":"D:\acr-test"` 的 `\a` 是非法转义,钩子解析失败就安全降级放行了。换成正斜杠,立刻就拦住了。

另一个更隐蔽。我往 `CLAUDE.md` 里写内容时用了标记 `<!-- BEGIN ai-coding-rules (managed) -->`。结果重装一次内容就重复一次,卸载也删不掉。盯着正则看半天才反应过来——括号 `(managed)` 在正则里是分组语法,不是匹配字面的括号。加了转义就好了。

修完重测：装两次内容只有一份,卸载干净,用户自己写的东西一点没动。

## 发布：和 npm 的几轮拉扯

代码过了,发布又遇到一串问题。`npm login` 登的是国内镜像源,只读,发不上去。改成官方源。2FA 拦路,建了两次 token 才搞定权限。包名 `ai-coding-rules` 跟已有的 `aicoding-rules` 太像被拒,加了命名空间 `@decklijia/ai-coding-rules` 才发布成功。

后来又接了 GitHub Actions 自动发布流水线,用 npm 的 Trusted Publishing + OIDC,不需要在 CI 里存 token。第一次跑报 404,排查发现是 npm 上填的组织名多了一个 `1`。改对之后,`0.1.4` 顺利由 CI 自动发出去了。

从整理到发布,前后大半天,一个连续会话完成。从 `0.1.0` 滚到 `0.1.8`,八个版本。

## 更进一步：它也能复制自己的生产方式

发布到 npm 的是工具本身——你用它部署我整理好的规则。但如果你想要自己的规则集,把仓库 clone 下来就行。

里面有一份 `AUTHORING-RULES.md`,写清楚了怎么从你自己的会话和文档中提炼规则,还附了一段可以喂给 agent 的 Runbook。你告诉它原始数据在哪,它就能按步骤提炼、归类、自检。自检步骤会扫描密钥泄露——因为原始配置里常常有真实的 API key,绝不能混进规则发出去。

这套"元生产"能力只留在 git 仓库里,不进 npm 包。fork 之后换成自己的身份就能用。

---

## 装上之后到底有什么变化

装完这套东西,你的 AI agent 会有几个明显的变化。

不再反复踩同一个坑。以前 Ant Design 按钮空格的问题、GORM 零值的问题、`git archive` 漏文件的问题,你踩过一次、告诉过 AI,下次换个项目它又忘了。现在这些知识跟着工具走,每个新会话都带着。

工程纪律变成了硬性约束,而不是口头提醒。你不用每次都跟 AI 说"部署前记得 commit"、"写完跑一下测试"。钩子会自动检查,不满足条件就拦住。这不是在请求 AI 配合,是绕过它的判断直接卡住。

技术栈知识按需出现。用 Ant Design 的时候自动加载 Ant Design 的注意事项,用 GORM 的时候加载 GORM 的。不用的栈不会出现在上下文里,不浪费 token。

如果你想试试：

```bash
npx @decklijia/ai-coding-rules install
```

源码和完整方法论在 [github.com/l17728/ai-coding-rules](https://github.com/l17728/ai-coding-rules)。拿去,改成你自己的,让你的 agent 从此带着记忆开工。
